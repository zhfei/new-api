# 不同渠道支持的接口和接口调用流程说明

本文用于说明 new-api 中“为什么不同渠道/模型支持的接口不一样”，以及请求进入系统后大致如何被路由、转换并转发到上游。

## 1. 为什么不同模型支持的接口不一样

不同模型支持的接口类型不一样，主要不是因为模型名字不同，而是因为“模型能力、上游协议、项目适配器”三层共同决定了最终可用接口。

| 层级 | 差异来源 | 例子 |
|---|---|---|
| 模型能力 | 模型本身定位不同 | 文本、图片、音频、向量、rerank、视频 |
| 上游 API 协议 | 厂商暴露的接口格式不同 | OpenAI Chat、OpenAI Responses、Claude Messages、Gemini GenerateContent |
| 项目适配器 | new-api 是否实现了对应转换 | 有的渠道能把 OpenAI 格式转成 Claude/Gemini，有的没有实现 |
| 账号体系 | 凭证和入口不同 | OpenAI API Key、ChatGPT/Codex OAuth、Gemini Key、Claude Key |
| 风控与计费 | 上游限制不同 | ChatGPT Plus/Codex 不等于 OpenAI Platform API |

可以简单理解为：

| 概念 | 类比 |
|---|---|
| 模型名 | 菜名 |
| 接口协议 | 点单系统 |
| 渠道适配器 | 服务员/翻译 |
| 上游厂商 | 后厨 |

同样叫 `gpt-5`，如果走 OpenAI Platform，可能使用 `/v1/chat/completions` 或 `/v1/responses`；如果走 Codex/ChatGPT 后端，则当前项目适配的是 `/backend-api/codex/responses`。

## 2. 外部接口到内部处理流程

外部请求首先进入 `router/relay-router.go`，再进入 `controller.Relay`，随后根据 `RelayMode` 分发到不同 helper。

| 外部接口 | RelayFormat | 主要 Helper | 用途 |
|---|---|---|---|
| `/v1/chat/completions` | `OpenAI` | `TextHelper` | 标准 OpenAI Chat Completions |
| `/v1/completions` | `OpenAI` | `TextHelper` | 旧 Completions |
| `/v1/responses` | `OpenAIResponses` | `ResponsesHelper` | OpenAI Responses API |
| `/v1/responses/compact` | `OpenAIResponsesCompaction` | `ResponsesHelper` | Responses 压缩接口，仅 OpenAI/Codex 类渠道支持 |
| `/v1/messages` | `Claude` | `ClaudeHelper` | Anthropic Messages 格式 |
| `/v1beta/models/*path` | `Gemini` | `GeminiHelper` / `GeminiEmbeddingHandler` | Gemini 原生格式 |
| `/v1/embeddings` | `Embedding` | `EmbeddingHelper` | 向量接口 |
| `/v1/images/generations` | `OpenAIImage` | `ImageHelper` | 图片生成 |
| `/v1/images/edits` | `OpenAIImage` | `ImageHelper` | 图片编辑 |
| `/v1/audio/transcriptions` | `OpenAIAudio` | `AudioHelper` | 语音转文字 |
| `/v1/audio/translations` | `OpenAIAudio` | `AudioHelper` | 语音翻译 |
| `/v1/audio/speech` | `OpenAIAudio` | `AudioHelper` | 文本转语音 |
| `/v1/rerank` | `Rerank` | `RerankHelper` | 重排 |
| `/mj/*` | Midjourney Task | MJ handler | Midjourney 异步任务 |
| `/suno/*` | Task | `RelayTask` / `RelayTaskFetch` | Suno 异步任务 |

## 3. 内部调用链

普通文本请求的大致流程如下。

| 阶段 | 关键代码 | 说明 |
|---|---|---|
| 路由匹配 | `router/relay-router.go` | 根据 URL 选择 RelayFormat |
| 鉴权与选渠道 | `middleware.TokenAuth`、`middleware.Distribute` | 校验 token，按模型/分组选择渠道 |
| 请求校验 | `helper.GetAndValidateRequest` | 解析请求体，识别模型和模式 |
| 生成上下文 | `relaycommon.GenRelayInfo` | 生成 RelayInfo，记录模型、渠道、用户、计费上下文 |
| 计费预扣 | `helper.ModelPriceHelper`、`service.PreConsumeBilling` | 估算 token，预扣额度 |
| Helper 分发 | `controller.relayHandler` | 根据 RelayMode 进入 Text/Responses/Claude/Gemini 等 helper |
| 适配器转换 | `relay/channel/*/adaptor.go` | 调用对应 `Convert*Request` |
| 上游请求 | `channel.DoApiRequest` 等 | 发请求到上游 |
| 响应处理 | `adaptor.DoResponse` | 转回统一响应格式，统计 usage |
| 后置计费 | `service.PostTextConsumeQuota` 等 | 根据真实 usage 结算 |

## 4. Adaptor 接口与支持能力

每个渠道是否支持某类接口，主要看该渠道 adaptor 是否实现了对应转换方法。

| Adaptor 方法 | 对应入口 | 说明 |
|---|---|---|
| `ConvertOpenAIRequest` | `/v1/chat/completions`、`/v1/completions` | OpenAI Chat/Completions 兼容入口 |
| `ConvertOpenAIResponsesRequest` | `/v1/responses`、`/v1/responses/compact` | Responses API 入口 |
| `ConvertClaudeRequest` | `/v1/messages` | Claude Messages 入口 |
| `ConvertGeminiRequest` | `/v1beta/models/*path` | Gemini 原生入口 |
| `ConvertEmbeddingRequest` | `/v1/embeddings` | Embedding |
| `ConvertImageRequest` | `/v1/images/*` | 图片 |
| `ConvertAudioRequest` | `/v1/audio/*` | 音频 |
| `ConvertRerankRequest` | `/v1/rerank` | Rerank |

如果某个 adaptor 返回 `not implemented` 或 `endpoint not supported`，说明这个渠道当前不能直接走对应接口。

## 6. Codex 渠道的特殊性

Codex 渠道使用的是 ChatGPT/Codex 后端，并不是 OpenAI Platform API Key 那套入口。

| 项目 | 当前行为 |
|---|---|
| 渠道类型 | `ChannelTypeCodex = 57` |
| 凭证类型 | OAuth JSON，包含 `access_token`、`refresh_token`、`account_id` 等 |
| Base URL | 默认 `https://chatgpt.com` |
| 支持入口 | `/v1/responses`、`/v1/responses/compact` |
| 上游路径 | `/backend-api/codex/responses`、`/backend-api/codex/responses/compact` |
| 不支持入口 | `/v1/chat/completions`、`/v1/messages`、Embedding、Image、Audio、Rerank、Gemini |

Codex key 的基本结构如下：

```json
{
  "access_token": "...",
  "refresh_token": "...",
  "account_id": "...",
  "email": "...",
  "type": "codex",
  "expired": "..."
}
```

当前 Codex adaptor 对 `/v1/chat/completions` 的行为是直接返回错误：

| 请求 | 当前结果 |
|---|---|
| `/v1/chat/completions` | `codex channel: /v1/chat/completions endpoint not supported` |
| `/v1/responses` | 正常转换并请求 Codex 后端 |

## 7. Chat Completions 到 Responses 的兼容转换

项目中已经有一套兼容机制，可以把 `/v1/chat/completions` 转成 `/v1/responses`。

| 条件 | 要求 |
|---|---|
| 全局请求体透传 | 必须关闭 |
| 渠道请求体透传 | 必须关闭 |
| 兼容策略 | `global.chat_completions_to_responses_policy` 必须命中 |
| 转换入口 | `chatCompletionsViaResponses` |
| 请求转换 | `service.ChatCompletionsRequestToResponsesRequest` |
| 响应转换 | `openai.OaiResponsesToChatHandler` / `openai.OaiResponsesToChatStreamHandler` |

建议用于 Codex 的策略示例：

```json
{
  "enabled": true,
  "all_channels": false,
  "channel_types": [57],
  "model_patterns": ["^gpt-.*$"]
}
```

该配置表示：

| 字段 | 含义 |
|---|---|
| `enabled` | 启用兼容转换 |
| `all_channels` | 不对所有渠道生效 |
| `channel_types: [57]` | 只对 Codex 渠道生效 |
| `model_patterns: ["^gpt-.*$"]` | 匹配所有 `gpt-` 开头模型 |

命中该策略后，用户仍然可以请求 `/v1/chat/completions`，项目内部会转换成 `/v1/responses` 再转发给 Codex。

## 8. 为什么有些“同名模型”接口也不同

同样叫 `gpt-5`，不同渠道背后可能完全不是同一个入口。

| 渠道 | 模型名示例 | 凭证 | 推荐接口 | 原因 |
|---|---|---|---|---|
| OpenAI Platform | `gpt-5` | OpenAI API Key | `/v1/chat/completions` 或 `/v1/responses` | 官方 Platform API |
| Codex/ChatGPT | `gpt-5`、`gpt-5-codex` | ChatGPT OAuth JSON | `/v1/responses` | ChatGPT/Codex 后端 |
| Claude | `claude-*` | Anthropic API Key | `/v1/messages` | Anthropic 原生 Messages |
| Gemini | `gemini-*` | Gemini API Key | `/v1beta/models/*path` | Google 原生 GenerateContent |

所以不能只看模型名判断接口，必须同时看渠道类型。

## 9. 排查接口不支持错误

遇到 `endpoint not supported` 或 `not implemented` 时，可以按下面顺序排查。

| 排查项 | 说明 |
|---|---|
| 当前请求 URL | 看客户端发的是 `/v1/chat/completions`、`/v1/responses` 还是其他 |
| 渠道类型 | 看选中的渠道是 OpenAI、Codex、Claude、Gemini 等 |
| 模型映射 | 看请求模型是否被映射到其他上游模型 |
| Adaptor 方法 | 看对应渠道是否实现 `Convert*Request` |
| 兼容策略 | Chat Completions 到 Responses 是否命中 |
| 透传开关 | 全局或渠道请求体透传开启时，部分转换不会执行 |
| 上游账号权限 | 即使项目支持，上游账号也可能没有对应模型或接口权限 |

## 10. Codex 场景的结论

| 目标 | 当前推荐做法 |
|---|---|
| 直接使用 Codex 原生能力 | 客户端请求 `/v1/responses` |
| 兼容普通 OpenAI Chat 客户端 | 配置 Chat Completions -> Responses 策略 |
| 让操练场测试 Codex 成功 | 确认操练场请求能命中转换策略，或改操练场支持 Responses |
| 长期产品化 | 可以考虑在代码中明确 Codex 是 Responses-only，并提供更友好的自动转换或 UI 提示 |
