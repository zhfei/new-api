# 一卡通支付宝当面付设计文档

## 1. 设计结论

本方案用于在 `new-api` 项目中接入支付宝官方当面付扫码支付，目标是同时支持“一卡通永久余额卡充值”和“日卡/周卡/月卡权益卡购买”。实现原则是：**复用当前已经稳定存在的 `TopUp`、`SubscriptionOrder`、支付设置、回调路由和履约逻辑，只新增支付宝当面付 Provider，不重做一套支付订单系统**。

| 项目 | 最终方案 |
|---|---|
| 支付方式 | 支付宝官方当面付扫码支付 |
| 支付接口 | `alipay.trade.precreate` |
| 签名方式 | `RSA2` |
| 支付页形态 | 后端生成二维码页面，前端内联二维码弹窗复用同一份支付数据 |
| 成功依据 | 只信任支付宝异步通知 `notify_url`，验签成功后履约 |
| 订单模型 | 钱包充值复用 `TopUp`；权益卡购买复用 `SubscriptionOrder` |
| 支付 Provider | 新增 `alipay_f2f`，与 `epay/stripe/creem/waffo` 并列 |
| 代码形态 | 只新增 `AlipayF2FProvider`，不重构现有 Epay 代码 |
| 接入范围 | 钱包充值、日卡/周卡/月卡权益卡购买 |
| 支付外显信息 | 支付宝侧使用合规安全标题和描述；平台内部保留真实套餐/权益信息 |

## 2. 当前 `new-api` 支付现状

`new-api` 当前支付体系已经具备完整的订单和履约基础。当面付只需要补“官方支付宝扫码通道”，不需要重写订单表。

| 支付场景 | 当前入口 | 订单表 | 回调入口 | 成功履约 |
|---|---|---|---|---|
| 钱包充值 | `POST /api/user/pay` | `topups` / `model.TopUp` | `/api/user/epay/notify` | `IncreaseUserQuota` 增加用户永久余额 |
| 权益卡购买 | `POST /api/subscription/epay/pay` | `subscription_orders` / `model.SubscriptionOrder` | `/api/subscription/epay/notify` | `CompleteSubscriptionOrder` 创建 `UserSubscription` |
| Stripe 充值 | `POST /api/user/stripe/pay` | `topups` | Stripe webhook | `model.Recharge` |
| Stripe 权益卡 | `POST /api/subscription/stripe/pay` | `subscription_orders` | Stripe webhook | `CompleteSubscriptionOrder` |
| Creem 权益卡 | `POST /api/subscription/creem/pay` | `subscription_orders` | Creem webhook | `CompleteSubscriptionOrder` |

当前关键代码：

| 文件 | 当前职责 |
|---|---|
| `controller/topup.go` | 钱包充值下单、Epay 回调、充值记录、补单 |
| `controller/subscription_payment_epay.go` | 权益卡 Epay 下单、回调、同步返回 |
| `model/topup.go` | `TopUp` 订单、支付网关防串单、充值履约 |
| `model/subscription.go` | `SubscriptionOrder`、权益卡支付完成、订单过期 |
| `service/epay.go` | 统一读取支付回调地址 |
| `setting/operation_setting/payment_setting_old.go` | Epay 地址、商户号、密钥、支付方式配置 |
| `web/default/src/features/wallet/hooks/use-payment.ts` | 钱包页发起支付并提交 Epay 表单 |
| `web/default/src/features/subscriptions/components/dialogs/subscription-purchase-dialog.tsx` | 权益卡购买弹窗和 Epay 表单提交 |

## 3. 当前项目已跑通的当面付方案可复用点

根目录项目中已经跑通的当面付实现采用了清晰的 Provider 抽象，可迁移到 `new-api`，但要用 Go 实现，不能照搬 TypeScript 代码。

| 已跑通能力 | 复用到 `new-api` 的方式 |
|---|---|
| `alipay.trade.precreate` 预下单 | 在 Go Provider 中实现同等请求参数和 RSA2 签名 |
| 支付宝返回 `qr_code` | 保存到 `TopUp` 或 `SubscriptionOrder` 的 provider payload 中，或返回给前端展示 |
| 二维码页轮询状态 | `new-api` 增加钱包和订阅各自的二维码页/状态接口，底层复用同一套查询函数 |
| 异步通知验签 | 钱包和订阅分别暴露 notify 路由，底层共用支付宝公钥验签逻辑 |
| 金额校验 | 钱包充值校验 `TopUp.Money`；权益卡校验 `SubscriptionOrder.Money` |
| 商户校验 | 校验 `app_id`，配置 `seller_id` 时校验 `seller_id` |
| 支付外显安全标题 | 支付宝上送固定安全标题/描述，内部订单仍保存真实业务信息 |
| Provider 多态 | `new-api` 新增 Go 版 `PaymentProvider` 接口和 `AlipayF2FProvider` 实现 |

## 4. 设计目标

| 目标 | 说明 |
|---|---|
| 简单 | 不新增支付订单表，不引入复杂支付编排系统 |
| 好用 | 管理端能配置，用户端能扫码，支付成功能自动到账 |
| 低侵入 | 当前 Epay、Stripe、Creem、Waffo 流程保持不变 |
| 可复用 | 钱包充值和权益卡购买共用同一个当面付 Provider |
| 可追踪 | 订单记录 `payment_provider=alipay_f2f`，日志能区分支付通道 |
| 安全 | 只信任支付宝异步通知，强校验签名、金额、AppID、SellerID |

## 5. 支付 Provider 抽象

当前 `new-api` 的 Epay 逻辑直接散落在 controller 中。本方案不重构 Epay，避免扩大改动面；当面付独立新增轻量 Provider，把 RSA2、二维码、验签逻辑封装起来，controller 只负责订单创建和履约调用。`PaymentProvider` 接口只服务当面付新增代码，`epay/stripe/creem/waffo` 保持当前实现方式，不迁移到新接口。

| 接口方法 | 作用 |
|---|---|
| `Type()` | 返回支付网关类型，如 `epay`、`alipay_f2f` |
| `DisplayName()` | 返回前端展示名称 |
| `ValidateConfig()` | 校验必要配置是否完整 |
| `CreatePayment(ctx)` | 创建支付订单，返回表单跳转或二维码支付信息 |
| `VerifyCallback(params)` | 验证回调签名 |
| `ParseCallback(params)` | 解析支付平台回调为统一结果 |

Go 接口形态：

```go
type PaymentProvider interface {
    Type() string
    DisplayName() string
    ValidateConfig() error
    CreatePayment(ctx PaymentCreateContext) (*PaymentCreateResult, error)
    VerifyCallback(params map[string]string) error
    ParseCallback(params map[string]string) (*PaymentCallbackResult, error)
}
```

统一数据结构：

| 结构 | 字段 | 说明 |
|---|---|---|
| `PaymentCreateContext` | `TradeNo` | 平台订单号，对应支付宝 `out_trade_no` |
| `PaymentCreateContext` | `Amount` | 支付金额，单位元，保留两位小数 |
| `PaymentCreateContext` | `Subject` | 支付宝订单标题 |
| `PaymentCreateContext` | `Body` | 支付宝订单描述 |
| `PaymentCreateContext` | `NotifyURL` | 支付宝异步通知地址 |
| `PaymentCreateContext` | `ReturnURL` | 兼容同步返回地址 |
| `PaymentCreateResult` | `PaymentPageURL` | 后端二维码页地址 |
| `PaymentCreateResult` | `QRCode` | 支付宝原始二维码内容 |
| `PaymentCreateResult` | `QRCodeDataURL` | 后端二维码页使用的图片；API 响应可只返回 `QRCode`，前端按需渲染 |
| `PaymentCreateResult` | `RawPayload` | 支付通道原始响应 |
| `PaymentCallbackResult` | `TradeNo` | 平台订单号 |
| `PaymentCallbackResult` | `ProviderTradeNo` | 支付宝交易号 |
| `PaymentCallbackResult` | `Status` | `success`、`pending`、`failed` |
| `PaymentCallbackResult` | `PaidAmount` | 支付宝回调金额 |
| `PaymentCallbackResult` | `AppID` | 支付宝应用 ID |
| `PaymentCallbackResult` | `SellerID` | 支付宝卖家 ID |

## 6. 支付配置项

在当前“支付设置”页面增加“支付宝当面付设置”卡片。配置继续走 `OptionMap`，避免新增配置表。

| 配置项 | 必填 | 默认值 | 说明 |
|---|---:|---|---|
| `AlipayF2FEnabled` | 是 | `false` | 是否启用支付宝当面付 |
| `AlipayF2FAppId` | 是 | 空 | 支付宝开放平台应用 AppID |
| `AlipayF2FPrivateKey` | 是 | 空 | 应用私钥，用于 RSA2 请求签名 |
| `AlipayF2FPublicKey` | 是 | 空 | 支付宝公钥，用于异步通知验签 |
| `AlipayF2FGatewayUrl` | 否 | `https://openapi.alipay.com/gateway.do` | 支付宝正式网关 |
| `AlipayF2FSandboxEnabled` | 否 | `false` | 是否使用沙箱网关 |
| `AlipayF2FTopUpNotifyUrl` | 否 | `<CallbackAddress>/api/user/alipay-f2f/notify` | 钱包充值异步通知地址 |
| `AlipayF2FTopUpReturnUrl` | 否 | `<CallbackAddress>/api/user/alipay-f2f/return` | 钱包充值同步返回地址 |
| `AlipayF2FSubscriptionNotifyUrl` | 否 | `<CallbackAddress>/api/subscription/alipay-f2f/notify` | 权益卡购买异步通知地址 |
| `AlipayF2FSubscriptionReturnUrl` | 否 | `<CallbackAddress>/api/subscription/alipay-f2f/return` | 权益卡购买同步返回地址 |
| `AlipayF2FSellerId` | 否 | 空 | 配置后强校验卖家 ID |
| `AlipayF2FMinTopUp` | 否 | `1` | 当面付最低充值金额 |
| `AlipayF2FDisplayName` | 否 | `支付宝当面付` | 前端展示名称 |

支付宝侧订单标题和描述不开放后台配置，代码中固定使用常量，避免管理员误配置导致支付侧外显信息不一致。

| 常量 | 固定值 | 用途 |
|---|---|---|
| `AlipayF2FSubject` | `启宝扫码点餐订单` | 支付宝侧固定展示订单标题，不使用真实套餐名 |
| `AlipayF2FBody` | `线下餐饮扫码点餐服务` | 支付宝侧固定展示订单描述，不使用真实业务描述 |

支付方式配置：

| 场景 | 配置方式 |
|---|---|
| 用户钱包充值展示 | `GetTopUpInfo` 在 `AlipayF2FEnabled=true` 且配置完整时追加 `pay_methods` 项 |
| 权益卡购买展示 | `SubscriptionPurchaseDialog` 复用 `pay_methods`，展示 `type=alipay_f2f` |
| Epay 继续可用 | `PayMethods` 中的 `alipay/wxpay/custom` 保持原逻辑 |
| 当面付独立可用 | `alipay_f2f` 不依赖 Epay 的 `PayAddress/EpayId/EpayKey` |

支付可用性返回规则：

| 返回字段 | 取值规则 | 用途 |
|---|---|---|
| `enable_online_topup` | 保持当前 Epay 含义，只表示 Epay 是否可用 | 兼容旧前端逻辑 |
| `enable_alipay_f2f_topup` | `payment_compliance_confirmed=true`、`AlipayF2FEnabled=true` 且 AppID、私钥、公钥、网关配置完整 | 前端判断当面付是否可展示 |
| `pay_methods` | Epay 支付方式和 `alipay_f2f` 都放在同一个数组中返回 | 前端统一展示支付方式 |
| 在线支付总开关 | 前端用 `enable_online_topup || enable_alipay_f2f_topup || enable_stripe_topup || enable_creem_topup || enable_waffo_topup` 判断 | 避免只开当面付时页面误判在线支付关闭 |

前端 `pay_methods` 示例：

```json
[
  {
    "name": "支付宝当面付",
    "type": "alipay_f2f",
    "color": "rgba(var(--semi-blue-5), 1)",
    "min_topup": "1"
  }
]
```

## 7. 支付宝请求字段映射

| 支付宝字段 | 来源 | 说明 |
|---|---|---|
| `app_id` | `AlipayF2FAppId` | 支付宝应用 ID |
| `method` | 固定 `alipay.trade.precreate` | 当面付预下单 |
| `format` | 固定 `JSON` | 响应格式 |
| `charset` | 固定 `utf-8` | 字符集 |
| `sign_type` | 固定 `RSA2` | 签名方式 |
| `timestamp` | 服务端当前时间 | 格式 `YYYY-MM-DD HH:mm:ss` |
| `version` | 固定 `1.0` | API 版本 |
| `notify_url` | 钱包充值用 `AlipayF2FTopUpNotifyUrl`，权益卡购买用 `AlipayF2FSubscriptionNotifyUrl` | 公网可访问 |
| `biz_content.out_trade_no` | `TopUp.TradeNo` 或 `SubscriptionOrder.TradeNo` | 平台订单号 |
| `biz_content.total_amount` | 订单金额 | 保留两位小数 |
| `biz_content.subject` | 支付安全标题 | 支付宝侧展示 |
| `biz_content.body` | 支付安全描述 | 支付宝侧描述 |
| `biz_content.product_code` | `FACE_TO_FACE_PAYMENT` | 当面付产品码 |

支付外显信息固定隔离：

| 层级 | 标题/描述 | 用途 |
|---|---|---|
| 支付宝侧 `subject` | 固定使用 `启宝扫码点餐订单` | 与已开通经营类目保持一致 |
| 支付宝侧 `body` | 固定使用 `线下餐饮扫码点餐服务` | 降低支付侧风控误伤 |
| 平台内部订单 | 真实充值金额、权益卡名称、套餐标题 | 后台排障、账单展示、履约 |

## 8. 钱包充值流程

钱包充值接入当面付时，继续复用 `TopUp` 表。

| 步骤 | 行为 | 代码落点 |
|---:|---|---|
| 1 | 用户在钱包页选择 `alipay_f2f` | `web/default/src/features/wallet` |
| 2 | 前端请求 `POST /api/user/alipay-f2f/pay` | 新增路由 |
| 3 | 后端校验支付合规确认、充值金额、配置完整性 | `controller/topup_alipay_f2f.go` |
| 4 | 计算支付金额 | 复用 `getPayMoney` |
| 5 | 生成 `TradeNo`，创建 `TopUp` 订单 | `PaymentProvider=alipay_f2f` |
| 6 | 调用 `AlipayF2FProvider.CreatePayment` | 新增 `service/payment/alipay_f2f.go` |
| 7 | 保存 `qr_code/raw_payload` | `TopUp` 增加 `ProviderPayload` 字段 |
| 8 | 前端展示二维码或打开二维码页 | 新增通用二维码支付展示 |
| 9 | 支付宝异步通知 `/api/user/alipay-f2f/notify` | 无需登录 |
| 10 | 验签、验金额、验 AppID/SellerID | Provider + controller |
| 11 | 订单加锁，确认 `TopUp` 为 pending | 复用 `LockOrder` |
| 12 | 更新 `TopUp.Status=success`，增加用户额度 | 复用 `IncreaseUserQuota` |
| 13 | 记录充值日志 | `RecordTopupLog(..., "alipay_f2f")` |

钱包当面付下单强制调用 `requirePaymentCompliance`，用户未确认支付合规条款时直接拒绝下单；旧 Epay 钱包充值保持当前逻辑，不因为当面付接入而改变。

钱包充值返回数据：

| 字段 | 说明 |
|---|---|
| `message` | `success` 或 `error` |
| `data.payment_provider` | `alipay_f2f` |
| `data.out_trade_no` | 平台订单号 |
| `data.qr_code` | 支付宝二维码原始内容 |
| `data.qr_code_data_url` | 后端二维码页使用；前端 API 可不依赖该字段 |
| `data.status_url` | 前端轮询支付状态 |
| `data.payment_page_url` | 后端二维码页地址 |

## 9. 权益卡购买流程

权益卡购买接入当面付时，继续复用 `SubscriptionOrder` 表，并保持 `CompleteSubscriptionOrder` 作为唯一履约入口。

| 步骤 | 行为 | 代码落点 |
|---:|---|---|
| 1 | 用户在权益卡弹窗选择 `alipay_f2f` | `SubscriptionPurchaseDialog` |
| 2 | 前端请求 `POST /api/subscription/alipay-f2f/pay` | 新增路由 |
| 3 | 后端校验套餐启用、购买上限、onecard 商品规则 | 复用当前订阅校验 |
| 4 | 创建 `SubscriptionOrder` | `PaymentProvider=alipay_f2f` |
| 5 | 调用 `AlipayF2FProvider.CreatePayment` | 生成二维码 |
| 6 | 前端展示二维码或打开二维码页 | 通用二维码支付展示 |
| 7 | 支付宝异步通知 `/api/subscription/alipay-f2f/notify` | 无需登录 |
| 8 | 订阅回调只查询 `SubscriptionOrder` | 保持和当前 Epay 订阅回调一致 |
| 9 | 验签、验金额、验 AppID/SellerID | Provider + controller |
| 10 | 调用 `CompleteSubscriptionOrder(tradeNo, payload, alipay_f2f, "alipay_f2f")` | 创建用户权益卡 |
| 11 | 前端轮询成功后刷新“我的权益卡” | 钱包页刷新订阅状态 |

权益卡当面付下单固定复用 `validateOneCardPlan`，并执行一卡通卡类商品硬校验：日卡、周卡、月卡必须 `total_amount > 0`；24 小时周期卡必须使用 `quota_reset_period=custom`、`quota_reset_custom_seconds=86400`；不满足规则的套餐禁止发起当面付支付。

当面付是独立支付通道，不依赖 Epay 的 `PayMethods`。`SubscriptionRequestAlipayF2F` 只校验 `AlipayF2FEnabled=true`、支付宝配置完整、套餐规则合法和用户购买限制；不调用 `operation_setting.ContainsPayMethod` 校验 `alipay_f2f`。

`CompleteSubscriptionOrder` 内部同步生成或更新订阅对应 `TopUp` 记录时，需要把 `SubscriptionOrder.PaymentProvider` 和 `SubscriptionOrder.ProviderPayload` 同步到 `TopUp.PaymentProvider` 和 `TopUp.ProviderPayload`。实现时先把当次回调的 `providerPayload` 写入 `order.ProviderPayload`，再调用 `upsertSubscriptionTopUpTx`；或把 `providerPayload` 作为参数传入同步函数，确保订阅订单和同步 `TopUp` 使用同一份当面付 payload。后台账单、支付渠道统计、排障审计统一用 `PaymentProvider=alipay_f2f` 识别当面付订单。

支付成功只完成订单和生成用户权益卡，不修改已有 API Key group。用户是否使用 `free`、`plus`、`pro`、`auto`，仍由用户在 API Key 创建或编辑时自行选择。

权益卡返回数据：

| 字段 | 说明 |
|---|---|
| `message` | `success` 或 `error` |
| `data.payment_provider` | `alipay_f2f` |
| `data.out_trade_no` | 订阅订单号 |
| `data.qr_code` | 支付宝二维码原始内容 |
| `data.qr_code_data_url` | 后端二维码页使用；前端 API 可不依赖该字段 |
| `data.status_url` | 支付状态轮询地址 |
| `data.payment_page_url` | 后端二维码页地址 |

## 10. 回调设计

当面付保持 `new-api` 当前支付风格：钱包充值和权益卡购买分别暴露回调路由，底层共用同一套 `AlipayF2FProvider` 验签、解析、金额校验 helper。这样路由清晰，订单查询简单，也避免统一入口里做复杂订单类型判断。

| 路由 | 方法 | 认证方式 | 挂载位置 | 作用 |
|---|---|---|---|---|
| `/api/user/alipay-f2f/pay` | `POST` | 用户登录 | `selfRoute` | 钱包充值当面付下单 |
| `/api/user/alipay-f2f/notify` | `GET/POST` | 无需登录 | `userRoute` | 钱包充值异步通知，验签后完成 `TopUp` |
| `/api/user/alipay-f2f/return` | `GET/POST` | 无需登录 | `userRoute` | 钱包充值同步返回，只做跳转，不作为成功依据 |
| `/api/user/alipay-f2f/qrcode` | `GET` | 用户登录 | `selfRoute` | 展示钱包充值支付二维码页 |
| `/api/user/alipay-f2f/status` | `GET` | 用户登录 | `selfRoute` | 前端轮询钱包充值订单状态 |
| `/api/subscription/alipay-f2f/pay` | `POST` | 用户登录 | `subscriptionRoute` | 权益卡购买当面付下单 |
| `/api/subscription/alipay-f2f/notify` | `GET/POST` | 无需登录 | `apiRouter` | 权益卡购买异步通知，验签后完成 `SubscriptionOrder` |
| `/api/subscription/alipay-f2f/return` | `GET/POST` | 无需登录 | `apiRouter` | 权益卡购买同步返回，只做跳转，不作为成功依据 |
| `/api/subscription/alipay-f2f/qrcode` | `GET` | 用户登录 | `subscriptionRoute` | 展示权益卡购买支付二维码页 |
| `/api/subscription/alipay-f2f/status` | `GET` | 用户登录 | `subscriptionRoute` | 前端轮询权益卡购买订单状态 |

回调处理顺序：

| 顺序 | 行为 |
|---:|---|
| 1 | 合并 Query 和 Form 参数 |
| 2 | 使用支付宝公钥 RSA2 验签 |
| 3 | 解析 `out_trade_no`、`trade_no`、`trade_status`、`total_amount`、`app_id`、`seller_id` |
| 4 | 非 `TRADE_SUCCESS` / `TRADE_FINISHED` 返回 `success` 并记录忽略日志 |
| 5 | 钱包回调只查 `TopUp`；订阅回调只查 `SubscriptionOrder` |
| 6 | 校验订单 `PaymentProvider == alipay_f2f` |
| 7 | 校验金额完全一致 |
| 8 | 校验 `app_id`，配置 `seller_id` 时校验 `seller_id` |
| 9 | 对订单号加锁 |
| 10 | pending 订单执行履约；非 pending 订单直接幂等返回 `success` |
| 11 | 写入订单状态、支付方式、完成时间和日志 |
| 12 | 返回支付宝要求的纯文本 `success` |

金额校验与 `new-api` 当前其他支付方式保持一致：钱包充值按 `TopUp.Money` 校验，权益卡购买按 `SubscriptionOrder.Money` 校验，沿用现有金额格式化和比较口径，不单独为当面付引入新的金额单位或精度规则。

回调成功状态：

| 支付宝 `trade_status` | 平台处理 |
|---|---|
| `TRADE_SUCCESS` | 支付成功，执行履约 |
| `TRADE_FINISHED` | 支付成功，执行履约 |
| 其他非空状态 | 不履约，记录日志并返回 `success` |
| 空状态 | 返回 `fail` |

## 11. 订单查询规则

为了保持实现简单，订单号继续沿用当前前缀规则，但不在统一回调中混合查表。钱包路由只处理钱包订单，订阅路由只处理订阅订单。

| 订单类型 | 当前/新增前缀 | 查询路由 | 表 |
|---|---|---|
| 钱包充值 | `USR{id}NO...` | `/api/user/alipay-f2f/notify` | `topups` |
| 权益卡购买 | `SUBUSR{id}NO...` | `/api/subscription/alipay-f2f/notify` | `subscription_orders` |

状态查询统一使用支付宝语义的 `out_trade_no` 参数，不再混用 `trade_no`。前端轮询地址固定形态为：`/api/.../alipay-f2f/status?out_trade_no=xxx`。

二维码页和状态查询必须做订单归属校验：钱包订单只允许订单所属用户查看自己的 `TopUp`；权益卡订单只允许订单所属用户查看自己的 `SubscriptionOrder`。如果 `order.user_id != 当前登录用户 id`，直接返回无权限，避免订单金额、状态和二维码信息泄露。

## 12. 数据库字段

当前 `TopUp` 和 `SubscriptionOrder` 已经有 `PaymentProvider` 字段，可以区分支付网关。`SubscriptionOrder.ProviderPayload` 当前已经存在，直接复用；`TopUp` 需要补充轻量 payload 字段。

| 表 | 字段 | 类型 | 说明 |
|---|---|---|---|
| `topups` | `provider_payload` | `TEXT` | 新增，保存当面付预下单响应和回调原文 |
| `subscription_orders` | `provider_payload` | `TEXT` | 已存在，保存当面付预下单响应和回调原文 |

本方案直接增加 `provider_payload` 字段，二维码页刷新、订单排障、回调审计都依赖该字段；不采用“只返回前端、不落库”的临时口径。

## 13. 前端设计

默认前端和经典前端都需要识别 `alipay_f2f`，但交互复用同一套二维码支付展示。

| 页面 | 改造点 |
|---|---|
| 支付设置页 | 新增“支付宝当面付设置”卡片，填写 AppID、私钥、公钥、网关、回调、沙箱、SellerID |
| 钱包充值页 | `pay_methods` 出现 `alipay_f2f` 时，点击后展示二维码弹窗 |
| 权益卡购买弹窗 | 支付方式列表包含 `alipay_f2f`，点击后展示二维码弹窗 |
| 二维码弹窗 | 展示金额、订单号、二维码、刷新按钮和支付状态 |
| 支付状态轮询 | 每 3 秒请求对应场景的 `/api/.../alipay-f2f/status?out_trade_no=...` |
| 支付成功 | 关闭弹窗，刷新钱包余额或我的权益卡 |
| 支付失败 | 显示错误，不自动履约 |

前端支付分流规则：

| 场景 | 固定处理 |
|---|---|
| 钱包充值选择 `alipay_f2f` | 调用 `POST /api/user/alipay-f2f/pay`，打开二维码弹窗，不调用 `/api/user/pay` |
| 权益卡购买选择 `alipay_f2f` | 调用 `POST /api/subscription/alipay-f2f/pay`，打开二维码弹窗，不调用 `/api/subscription/epay/pay` |
| `alipay_f2f` 出现在 `pay_methods` | 只作为展示数据复用，不进入 Epay 表单提交逻辑 |
| 状态轮询 | 使用后端返回的 `status_url`，参数名固定为 `out_trade_no` |
| 二维码展示 | 优先使用后端返回的 `payment_page_url`，前端弹窗可直接嵌入该地址；需要内联二维码时再使用 `qr_code` 渲染 |

支付展示策略：

| 支付类型 | 展示方式 |
|---|---|
| `epay` | 保持当前 POST 表单跳转 |
| `stripe` | 保持打开 checkout 链接 |
| `creem` | 保持打开 checkout 链接 |
| `waffo` | 保持当前专用流程 |
| `alipay_f2f` | 二维码弹窗或后端二维码页 |

## 14. 安全规则

| 规则 | 固定处理 |
|---|---|
| 验签 | 必须使用支付宝公钥 RSA2 验签，排除 `sign` 和 `sign_type` 后排序 |
| 金额 | 回调 `total_amount` 必须等于平台订单金额 |
| AppID | 回调 `app_id` 必须等于 `AlipayF2FAppId` |
| SellerID | 配置后必须等于回调 `seller_id` |
| 订单网关 | 订单 `PaymentProvider` 必须是 `alipay_f2f` |
| 幂等 | 非 pending 成功订单直接返回 `success` |
| 浏览器 return | 只跳转页面，不做支付成功依据 |
| 本地域名 | `notify_url` 不能是 `localhost`、`127.0.0.1`、`0.0.0.0` |
| 私钥格式 | 支持 PEM 和纯 key，服务端统一规范化 |
| 日志 | 不打印私钥、公钥完整内容，不打印敏感配置 |

## 16. 实施顺序

| 顺序 | 任务 | 验收标准 |
|---:|---|---|
| 1 | 后端 Provider 抽象和 `AlipayF2FProvider` | 能生成合法 RSA2 请求并拿到 `qr_code` |
| 2 | 钱包充值下单和钱包回调 | 扫码后 `TopUp` 成功，用户永久余额增加 |
| 3 | 权益卡购买下单和订阅回调 | 扫码后 `SubscriptionOrder` 成功，生成 `UserSubscription` |
| 4 | 二维码页和状态轮询 API | 用户能扫码并自动看到成功状态 |
| 5 | 默认前端支付设置和钱包/权益卡二维码弹窗 | 管理员能配置，用户能购买 |
| 6 | 经典前端同步适配 | 两套前端能力一致 |
| 7 | 测试覆盖 | 签名、验签、金额校验、网关防串单、幂等回调均有测试 |

## 17. 测试清单

| 测试项 | 覆盖点 |
|---|---|
| RSA2 签名生成 | 参数排序、私钥 PEM 规范化 |
| 支付宝回调验签 | 排除 `sign/sign_type`，公钥 PEM 规范化 |
| 钱包充值成功 | `TopUp` pending -> success，用户 quota 增加 |
| 权益卡购买成功 | `SubscriptionOrder` pending -> success，创建权益卡 |
| 金额不一致 | 拒绝履约，返回 `fail` |
| AppID 不一致 | 拒绝履约，返回 `fail` |
| SellerID 不一致 | 配置 SellerID 后拒绝履约 |
| 支付网关不匹配 | Epay 订单不能被当面付回调完成 |
| 重复回调 | 第二次回调幂等返回 `success`，不重复加额度 |
| 非成功状态 | 不履约，记录日志 |
| 配置缺失 | 下单时返回明确错误 |
| 本地回调地址 | 阻止使用 localhost 作为 notify_url |
| 日/周/月卡规则 | 当面付下单复用 `validateOneCardPlan`，卡类商品规则不合法时拒绝支付 |
| API Key group 不变 | 购买权益卡成功后不修改已有 API Key group |
| 一卡通扣费闭环 | 支付成功生成 `UserSubscription` 后，请求命中 `SubscriptionFirstGroups` 时优先扣权益卡 |

## 18. 与 Epay 的关系

| 对比项 | Epay | 支付宝当面付 |
|---|---|---|
| 类型 | 聚合支付 | 支付宝官方直连 |
| 签名 | MD5 | RSA2 |
| 下单方式 | 提交第三方支付表单 | 调 `alipay.trade.precreate` 获取二维码 |
| 回调验签 | Epay Key | 支付宝公钥 |
| 支付方式字段 | `alipay`、`wxpay`、`custom` | 固定 `alipay_f2f` |
| 配置依赖 | `PayAddress/EpayId/EpayKey` | `AlipayF2FAppId/PrivateKey/PublicKey` |
| 订单履约 | 当前逻辑 | 复用当前逻辑 |

## 19. 最终流程图

```text
管理员配置支付宝当面付
  -> 用户购买永久余额卡或日/周/月权益卡
  -> new-api 创建 TopUp 或 SubscriptionOrder
  -> AlipayF2FProvider 调用 alipay.trade.precreate
  -> 返回 qr_code
  -> 前端展示二维码
  -> 用户支付宝扫码付款
  -> 支付宝异步通知钱包或订阅对应 notify 路由
  -> RSA2 验签
  -> 校验金额、AppID、SellerID、PaymentProvider
  -> 按订单类型履约
  -> 钱包充值增加 users.quota / 权益卡创建 UserSubscription
  -> 前端轮询成功并刷新页面数据
```

## 20. Review 结论

本方案是对 `new-api` 当前支付体系的低侵入增强：**当面付只作为新的支付 Provider 接入，订单、履约、日志、补单、权益卡闭环继续复用现有模型**。这样开发量最小，风险最低，也最容易和当前一卡通业务闭环合并。
