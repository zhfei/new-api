# 一卡通 429 自动禁用与自动恢复方案

## 1. 目标

| 项目 | 说明 |
|---|---|
| 核心目标 | 渠道返回 `429` 后，统一走现有“禁用”机制 |
| 恢复目标 | 到达恢复时间后，系统自动恢复渠道 |
| 设计原则 | 简单、可复用、低侵入 |
| 设计范围 | 按项目现有结构设计，统一使用 `channels.other_info` 解决 429 自动恢复 |

一句话：**429 直接禁用，记录原因和恢复时间；定时器每 30 秒扫描，到期自动启用。**

## 2. 当前问题

| 问题 | 说明 |
|---|---|
| 429 默认不会禁用 | 当前项目默认自动禁用状态码只有 `401` |
| 429 虽然可重试 | 但默认 `RetryTimes = 0` 时不会真的换渠道 |
| 下次还可能选中 | 429 渠道仍是 enabled，后续请求可能继续撞上 |
| 人工处理成本高 | 需要管理员手动禁用/启用 |

## 3. 简化后的核心规则

| 场景 | 处理 |
|---|---|
| 渠道返回 `429` | 直接禁用该渠道 |
| 禁用状态 | 使用现有 `ChannelStatusAutoDisabled` |
| 禁用原因 | 写入渠道数据库，标记为 429 限流 |
| 恢复时间 | 写入渠道数据库 |
| 定时扫描 | 每 30 秒扫描一次被禁用渠道 |
| 到期恢复 | 如果是 429 导致禁用，且达到恢复时间，自动启用 |
| 非 429 禁用 | 不自动按 429 规则恢复 |

## 4. 数据存储

429 自动恢复统一使用项目已有的 `channels.other_info`。该字段当前已经用于记录 `status_reason`、`status_time` 等渠道状态信息，因此继续在这里扩展 429 自动恢复信息，避免新增表和额外一致性问题。

| 字段 | 类型 | 示例 | 说明 |
|---|---|---|---|
| `status_reason` | string | `onecard_429_auto_disabled` | 人类可读的禁用原因 |
| `status_time` | int64 | `1716547200` | 禁用发生时间，Unix 秒 |
| `onecard_auto_recover_reason` | string | `429` | 自动恢复原因类型 |
| `onecard_auto_recover_at` | int64 | `1716565200` | 计划恢复时间，Unix 秒 |
| `onecard_auto_recover_model` | string | `gpt-5.4` | 触发 429 的模型 |
| `onecard_auto_recover_group` | string | `plus` | 触发 429 的实际池组 |
| `onecard_auto_recover_error` | string | `rate limit reached` | 最近一次 429 错误摘要 |

关键约束：

| 约束 | 说明 |
|---|---|
| 权威数据源 | `channels.other_info` 就是 429 自动恢复的权威数据源 |
| 不新增表 | 不新增 `onecard_channel_recoveries` 等专用表 |
| 查询方式 | 定时器扫描 `ChannelStatusAutoDisabled` 渠道，再解析 `other_info` 判断是否到期 |
| 前端提示 | 渠道列表直接根据 `other_info` 判断是否将名称标黄 |
| 跨数据库兼容 | 不新增迁移，继续使用现有 `channels.other_info` TEXT 字段 |

## 5. 429 恢复时间规则

| 错误内容 | 恢复时间 |
|---|---:|
| 包含 `5h`、`5 hours`、`try again in 5 hours` | 当前时间 + 5 小时 |
| 包含 `7d`、`7 days`、`weekly limit` | 当前时间 + 7 天 |
| 无法识别具体时长 | 当前时间 + 5 小时 |

本方案先内置上面三类核心规则，后续如果发现新的上游文案，再补解析规则。

## 6. 请求处理流程

| 步骤 | 行为 |
|---|---|
| 1 | 用户使用 API Key 请求 `free`、`plus`、`pro` 或 `auto` |
| 2 | 系统选择一个渠道并请求上游 |
| 3 | 如果上游返回非 `429`，继续走现有错误处理 |
| 4 | 如果上游返回 `429`，判断该渠道是否允许自动禁用 |
| 5 | 写入禁用原因和恢复时间 |
| 6 | 调用现有渠道禁用逻辑，将渠道置为自动禁用 |
| 7 | 对应 `abilities.enabled=false`，后续请求不会再选中 |
| 8 | 如果本次请求还有重试次数，继续换其他渠道 |
| 9 | 如果没有重试次数，向用户返回错误 |

## 7. 定时恢复流程

| 步骤 | 行为 |
|---|---|
| 1 | 启动 OneCard 429 自动恢复定时器 |
| 2 | 每 30 秒扫描一次渠道列表 |
| 3 | 查询状态为 `ChannelStatusAutoDisabled` 的渠道 |
| 4 | 解析每个渠道的 `other_info` |
| 5 | 只处理 `onecard_auto_recover_reason = 429` 的渠道 |
| 6 | 判断 `onecard_auto_recover_at <= now` |
| 7 | 如果渠道仍是自动禁用，则恢复渠道为 enabled |
| 8 | 恢复对应 `abilities.enabled=true` |
| 9 | 清理 `onecard_auto_recover_*` 字段并写入系统日志 |

## 8. 和现有禁用逻辑的关系

| 逻辑 | 是否复用 |
|---|---:|
| `ChannelStatusAutoDisabled` | 复用 |
| `service.DisableChannel` | 尽量复用 |
| `service.EnableChannel` | 复用 |
| `model.UpdateChannelStatus` | 复用 |
| `abilities.enabled` 更新 | 复用现有状态更新逻辑 |
| 现有自动禁用关键词 | 保留 |
| 现有自动禁用状态码 | 保留 |

## 9. 为什么这样更简单

| 方案 | 复杂度 | 说明 |
|---|---:|---|
| 独立冷却状态 | 高 | 需要额外区分 cooling、disabled、manual disabled |
| 429 统一走禁用 | 低 | 复用现有渠道状态和能力表 |
| 30 秒扫描恢复 | 低 | 逻辑清晰，容易排查 |
| 统一使用 `other_info` | 低 | 不新增表，不存在双写一致性问题，前端也能直接提示 |

## 10. 边界规则

| 场景 | 处理 |
|---|---|
| 手动禁用渠道 | 不自动恢复 |
| 非 429 自动禁用渠道 | 不按 429 规则恢复 |
| 429 禁用期间管理员手动启用 | 尊重管理员操作 |
| `onecard_auto_recover_at` 缺失 | 不自动恢复，避免误启用 |
| 恢复后再次 429 | 再次禁用并刷新恢复时间 |
| `auto` 请求命中 `plus` 渠道 | 禁用实际命中的渠道 |
| 多 key 渠道 | 沿用现有多 key 禁用逻辑；如需 key 级恢复，可在 `other_info` 中增加 key 维度结构 |

## 11. 模块设计

| 模块 | 职责 |
|---|---|
| `pkg/onecard/recover.go` | 429 判断、恢复时间解析、元数据封装 |
| `service/onecard_recover.go` | 禁用渠道、恢复渠道 |
| `service/onecard_recover_task.go` | 每 30 秒扫描并恢复到期渠道 |
| `controller/relay.go` | 在渠道错误处理处增加 OneCard 429 hook |
| `model/channel.go` | 复用现有状态更新能力 |

## 12. 面向对象策略

| 类型 | 职责 |
|---|---|
| `AutoRecoverPolicy` | 定义是否可自动恢复、恢复时间、恢复原因 |
| `BaseAutoRecoverPolicy` | 默认处理状态码和默认恢复时间 |
| `Codex429RecoverPolicy` | 解析 Codex / ChatGPT 的 5h、7d 限制 |
| `NoopRecoverPolicy` | 不支持自动恢复的渠道直接跳过 |

接口示例：

```go
type AutoRecoverPolicy interface {
    Match(channel ChannelInfo) bool
    ShouldAutoDisable(ctx *RequestContext, err *types.NewAPIError) bool
    RecoverAt(ctx *RequestContext, err *types.NewAPIError, now int64) int64
    Reason(ctx *RequestContext, err *types.NewAPIError) string
}
```

## 14. 推荐落地顺序

| 阶段 | 目标 |
|---|---|
| M1 | 新增 OneCard 429 自动恢复策略 |
| M2 | 429 错误时写入 `other_info` 并禁用渠道 |
| M3 | 新增 30 秒定时扫描任务 |
| M4 | 到期调用现有启用逻辑恢复渠道，并清理 `other_info` 自动恢复字段 |
| M5 | 增加单元测试和并发恢复测试 |
| M6 | 前端展示恢复原因、计划恢复时间、恢复状态 |

## 15. 总结

| 结论 | 说明 |
|---|---|
| 429 统一走禁用 | 渠道不会继续被选中 |
| 禁用不是永久的 | 429 禁用会带恢复时间 |
| 定时器自动恢复 | 每 30 秒扫描，到期自动启用 |
| 不新增表 | 统一使用已有 `channels.other_info` 作为恢复任务权威数据源 |
| 逻辑封装在 OneCard | 主链路只增加少量 hook |
